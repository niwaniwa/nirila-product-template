# レビュー指針ガイド

> レビュー指摘パターンの蓄積方法と、AI×人間のレビュー運用ルール

## 概要

レビュー指針は「過去の指摘パターンを蓄積し、次回以降のレビューに反映する」仕組み。
AIが機械的チェックを担い、人間は設計判断とビジネスロジックに集中する。

## ファイル構成

| ファイル | 役割 | 読み取りタイミング |
|---|---|---|
| `CLAUDE.md` | 全タスク共通の指示（レビュー時も参照） | 常時 |
| `REVIEW.md`（リポジトリルート） | レビュー専用ガイダンス | コードレビュー時のみ |
| `docs/guides/review-guidelines.md` | レビュー運用ルール（本ファイル） | 人間が参照 |

---

## REVIEW.md テンプレート

リポジトリルートに `REVIEW.md` を配置する。Claude Code のCode Review機能はこのファイルを自動検出する。

```markdown
# Code Review Guidelines

## Always check
<!-- プロジェクト固有の必須チェック項目を蓄積 -->
-

## Style
<!-- コーディングスタイルのルール -->
-

## Skip
<!-- レビュー不要なファイル・パターン -->
- _template.md files under docs/
```

### 記載する内容

- **組織・プロジェクト固有のパターン**: AIが一般知識から推測できないルール
- **過去のインシデントから得た教訓**: 「XXの変更時はYYも確認」
- **フレームワーク固有の規約**: プロジェクトで採用した特定のパターン

### 記載しない内容

- 一般的なコーディング規約（AIが既知の知識として保有）
- フォーマットルール（Hooksで自動処理すべき）
- 型チェックやlintで検出できる項目（CIで自動化すべき）

## AIと人間の役割分担

### AIが担当するレビュー

- ガイドライン違反の検出
- セキュリティ上の懸念（secrets漏洩、SQLインジェクション等）
- CIエラー対応の確認
- レビューコメント対応の抜け漏れ検証
- コード品質メトリクス（複雑度、重複等）

### 人間が担当するレビュー

- 設計判断の妥当性（ADRとの整合性）
- ビジネスロジックの正確性
- ユーザー体験への影響評価
- マージ可否の最終判定

---

## 指摘パターンの蓄積フロー

```
レビューで指摘発生
  ↓
指摘が繰り返し発生するパターンか判断
  ↓ Yes
REVIEW.md に追記（AIチェック対象化）
  or
Hooksに自動修正ルール追加（機械的に修正可能な場合）
  or
CLAUDE.md に追記（実装時から防止すべき場合）
```

### 蓄積の判断基準

| 条件 | 追加先 |
|---|---|
| レビューでのみ検出すべき観点 | `REVIEW.md` |
| 実装時から守るべきルール | `CLAUDE.md` |
| 機械的に自動修正可能 | Hooks設定 |
| lint/formatで対応可能 | CI設定 |

## 運用のポイント

- **REVIEW.mdは150〜200指示以内に保つ**: フロンティアLLMの一貫性が維持できる上限
- **チェックリスト形式を優先**: リスト化しないとAIは1〜2点に集中して他を無視する
- **定期的な棚卸し**: 不要になったルールを削除し、常にシンプルに保つ
- **具体例を添える**: 「早期リターンを優先」より「条件Xの場合はガード節で返す」が効果的
