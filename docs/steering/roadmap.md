## 開発モデル

本プロジェクトは **Water-Scrum-Fall** ハイブリッドモデルを採用する。
全体構造はウォーターフォール（要求整理→仕様策定→設計→実装→レビュー→リリース）で管理し、実装フェーズ内はイテレーティブに進める。

| フェーズ | 手法 | 進め方 |
|---|---|---|
| 要求整理 | Waterfall | コアバリュー・ターゲット・MVPを確定。手戻り最小化 |
| 仕様策定 | Waterfall + Agile | spec全体はWaterfallで固め、詳細はイテレーティブに精査 |
| 設計 | Waterfall | ADRで意思決定を固定。判断根拠を記録 |
| 実装 | Agile | spec単位で1-2週間イテレーション。TDD推奨 |
| レビュー | Agile | 各イテレーション末にレビュー・フィードバック |
| リリース | Waterfall | チェックリストに沿って段階的にリリース |

## 優先度付けフレームワーク

specの `phase` 割り当て時に以下を判断軸とする：

### MoSCoW法

| 分類 | 基準 | 対応フェーズ |
|---|---|---|
| **Must** | なければプロダクトが成立しない | MVP |
| **Should** | あれば価値が大幅に向上する | Phase 1 |
| **Could** | あれば良いが必須ではない | Phase 2 |
| **Won't** | 今回は対象外 | Backlog |

### 判断の観点

- **ユーザー価値**: コアバリューに直結するか
- **技術リスク**: 未検証技術を含むか（含む場合はPoCを先行）
- **依存関係**: 他機能の前提条件になっているか
- **開発コスト**: 工数に対するインパクトの比率

## spec との紐付けルール

- specテンプレートの `phase` フィールドに上記フェーズ名を記入
- 1つのspecが複数フェーズにまたがる場合は分割を検討
- フェーズ変更時は理由をADRに記録

---

## イテレーション運用（実装フェーズ）

小規模・個人開発では以下を目安とする：

- **1イテレーション**: 1〜2週間
- **単位**: accepted な spec 1つ（または関連するspec群）
- **サイクル**: 実装 → テスト → レビュー → 振り返り
- **振り返り**: レビュー指針の更新、フィードバックループの改善に反映
